Research date: 2026-07-22

## Source pin and method

This report statically inspected the official Joplin repository at commit [`5df269104ef233d78c535e9df8abd1061c49233c`](https://github.com/laurent22/joplin/tree/5df269104ef233d78c535e9df8abd1061c49233c) (2026-07-22, current `dev` tip when inspected). The bundled Backup plugin was checked at its upstream commit [`6a192515eaf71fa6f1aaaa74d406dae45981aa42`](https://github.com/JackGruber/joplin-plugin-backup/tree/6a192515eaf71fa6f1aaaa74d406dae45981aa42). Official Joplin and Electron documentation is used where it owns the behavior.

Confidence labels in this report mean:

- **Confirmed**: a named path and read/write lifecycle are explicit in source.
- **Inferred**: the path or security consequence follows from the implementation, but its exact runtime contents were not observed.
- **Trace**: static analysis cannot bound the contents; the encryption prototype must capture filesystem activity at runtime.

This was a static investigation. It did not build or run Joplin, so the runtime-tracing section is a required follow-up rather than optional polish.

## Executive conclusion

Stock Joplin's E2EE is a sync-transport boundary, not local-at-rest encryption. Joplin's own specification says items are encrypted only when `BaseItem.serializeForSync` serialises them for the sync target and are later decrypted in the background ([Joplin E2EE specification](https://joplinapp.org/help/dev/spec/e2ee/)). The live desktop profile deliberately contains decrypted notes, derived search copies, history, conflict bases, attachments, settings and plugin data.

A database-only encryption change is therefore insufficient for Watchtower One's stated “whole user-data profile” guarantee. At minimum the boundary must cover the active profile's SQLite database and sidecars, `resources`, `plugin-data`, Electron's `internal` session, settings, histories, indexes, logs, temporary/editor copies and backup behavior. Several high-risk spill sites sit outside the profile entirely: crash reports, the default Backup plugin destination, user-selected exports and any file opened in an external editor.

The most consequential startup fact is that Joplin reads `settings.json`, creates log targets, creates its Electron session and opens profile locks before or alongside opening `database.sqlite`. Watchtower's unlock gate must therefore run in the Electron main process before Joplin initialises any of those stores; wrapping only the database open is too late.

## Profile topology

`determineProfileAndBaseDir` selects either an explicit `--profile` directory, a portable `JoplinProfile` beside the executable, or `<home>/.config/<appName>`; alternate instances append their ID ([source](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/determineBaseAppDirs.ts)). Joplin's installation documentation independently confirms that Windows portable builds create `JoplinProfile` beside the executable ([official installation documentation](https://joplinapp.org/help/install/)).

The selected location is a **root profile directory**. `profiles.json` chooses the current profile; the default profile lives at the root while additional profiles live under `profile-<id>` ([profile initialisation](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/profileConfig/initProfile.ts), [profile path rules](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/profileConfig/index.ts#L45-L91)). Global settings and installed plugin packages are shared at the root, while databases, resources and plugin data are per-profile. Consequently, encrypting only `profile-<id>` would still expose root-level profile names, global settings, layout files, logs and plugin packages.

At startup `BaseApplication` assigns these paths and creates them:

```text
<root>/
  profiles.json
  settings.json                 # default profile and global settings
  plugins/                      # installed .jpl packages (shared)
  profile-<id>/                 # additional profile, when selected
  <active-profile>/
    database.sqlite
    resources/
    plugin-data/
    tmp/
    cache/
```

The constants and database open are explicit in [`BaseApplication.ts`](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/BaseApplication.ts#L738-L823). `settings.json` may exist both at the root and in a sub-profile; Joplin merges global root settings with local profile settings ([`Setting.ts`](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/models/Setting.ts#L350-L383), [`Setting.saveAll`](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/models/Setting.ts#L1169-L1208)).

## What `database.sqlite` exposes

The database is not merely the canonical note store. It contains multiple plaintext or user-derived projections:

1. **Canonical items and metadata.** `notes` includes title, body, author, source URL, geolocation, application/user data and timestamps; folders/tags contain titles and user data; `resources` contains titles, original filenames, MIME types and OCR text/details. The generated current schema lists these fields directly ([schema](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/database/types.ts#L412-L557), [notes](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/database/types.ts#L597-L630)).
2. **History and recently replaced text.** `revisions` stores title/body/metadata diffs. Before revision collection, `item_changes.before_change_item` can contain the complete prior note JSON ([note save](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/models/Note.ts#L830-L893), [revision collector](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/RevisionService.ts#L116-L185)). Deleting note history explicitly clears this cache, confirming it is sensitive history rather than harmless bookkeeping ([source](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/RevisionService.ts#L372-L390)).
3. **Search duplicates.** `notes_normalized` and its FTS4 shadow tables duplicate note title/body and metadata. The newer `items_normalized`/`items_fts` additionally index resource OCR title/body ([search migration](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/database/migrations/45.ts)).
4. **AI-derived content.** `note_embeddings_meta.chunk_text` stores plaintext note chunks and the lazily-created `note_embeddings_vec` stores their vectors; Joplin explicitly says these are local and not synced ([migration](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/database/migrations/52.ts), [vector table](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/models/NoteEmbedding.ts#L60-L70)). Embeddings should be treated as sensitive derived data even though reconstructing exact text from a vector is not equivalent to reading `chunk_text`.
5. **Sync conflict copies.** Current `sync_items` rows include `base_title` and `base_body`, and `conflict_note_states` contains local base plus remote title/body ([schema](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/database/types.ts#L466-L482), [conflict schema](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/database/migrations/53.ts)).
6. **Plugin state.** Plugin settings are namespaced and default to database storage unless the plugin requests file storage; a plugin may mark a setting secure ([plugin settings API](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/plugins/api/JoplinSettings.ts#L91-L123)). Plugins may also attach arbitrary synced `user_data` to notes, folders, tags and resources, which is stored in the corresponding item row.
7. **Key and credential material.** E2EE master keys, note-lock keys and the public/private key pair are serialised into `syncInfoCache`; master/private key contents are password-encrypted but remain high-value material ([`SyncInfo`](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/synchronizer/syncInfoUtils.ts#L292-L350)). Secure settings include sync passwords, AI API keys and the E2EE password cache ([representative settings](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/models/settings/builtInMetadata.ts#L190-L250), [password cache](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/models/settings/builtInMetadata.ts#L1540-L1556)).

SQLite can create rollback journals, WAL/SHM files and temporary files depending on driver/runtime settings. The inspected code does not explicitly force a journal mode, so exact filenames and lifetime are **Trace**, not a justified static claim. Any database encryption prototype must kill the process during a large write, inspect sidecars, and verify no note signatures survive in allocated files, deleted files or the filesystem journal.

## Credential handling is helpful but not a vault boundary

Joplin checks the settings table first. A secure setting is moved to the keychain only when the keychain write succeeds; on failure it is intentionally retained in plaintext in the settings table to avoid losing the user's password ([load precedence](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/models/Setting.ts#L643-L718), [save fallback](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/models/Setting.ts#L1120-L1164)).

On desktop the “keychain” driver calls Electron `safeStorage`, stores the resulting base64 ciphertext in the database's `key_values` table, and decrypts it through Electron on read ([Joplin driver](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/lib/services/keychain/KeychainServiceDriver.electron.ts)). Electron documents that synchronous `safeStorage` uses DPAPI on Windows, Keychain on macOS and desktop-dependent secret stores on Linux; Windows protects against other users but not other applications running as the same user ([Electron documentation](https://www.electronjs.org/docs/latest/api/safe-storage)).

Implications:

- SQLCipher/full-vault encryption is still required even if every `secure` setting successfully uses `safeStorage`.
- Watchtower's vault key and Joplin sync keys must remain domain-separated; reusing Electron `safeStorage` as the whole-vault root would not meet a strong same-user malware/offline-copy threat model.
- Keychain failure must become an explicit secure failure state for Watchtower rather than silently downgrading secrets to a plaintext database unless that database is already inside the unlocked encrypted vault.

## Architecture consequences for the encryption prototype

These are evidence-based constraints, not a final architecture decision:

1. **Disfavor “SQLCipher only.”** SQLCipher would cover the canonical database, histories, FTS, embeddings and credential fallback, but not resource blobs, settings JSON, plugin data, Electron session state, logs, temp files, external editor copies, crash reports or backups.
2. **Gate before Joplin bootstrap.** `main.ts` reads root `settings.json` to decide crash-upload behavior before constructing `ElectronAppWrapper` ([source](https://github.com/laurent22/joplin/blob/5df269104ef233d78c535e9df8abd1061c49233c/packages/app-desktop/main.ts#L43-L79)). The vault must be selected and unlocked before that read, before the Electron session at `internal`, before file log targets, and before `database.sqlite`.
3. **Treat the root profile as the natural data unit.** Multi-profile support, global settings and shared plugin packages cross active-profile boundaries. A per-subprofile vault requires a deliberately split launcher manifest and plugin-code store; encrypting the entire root is simpler and harder to misconfigure.
4. **Separate immutable plugin code from persistent plugin data.** Watchtower may keep signed curated `.jpl` packages outside the vault, but `plugin-data`, plugin settings and item `user_data` remain inside. Because desktop plugins run with powerful filesystem access, signing/curation is a trust decision, not technical confinement; runtime tests must attempt writes outside the assigned data directory.
5. **Replace stock Backup before enabling it.** The default plugin creates recoverable plaintext copies outside the profile. The first Watchtower build should ship it disabled unless an encrypted, authenticated, recoverable backup format is already wired to the independent recovery secret.
6. **Define explicit egress.** Export, print, external note editing and external attachment editing inherently produce plaintext for another application. They need a product-level warning and possibly policy controls; no local encryption design can truthfully claim those third-party files remain protected.
7. **Own crash behavior.** Redirecting normal logs into the vault does not stop OS/Sentry crash dumps from escaping. Pre-unlock and crash logging should be structured, content-minimised and tested with unique canary note text.
8. **Lock is more than key disposal.** On lock/exit, Watchtower must close SQLite, terminate/wait for plugin and renderer processes, stop external watchers, clean temp/edit copies, unmount or close the encrypted store, and then discard keys. A process kill during each stage is part of the acceptance test.

The evidence therefore favors prototyping a whole-profile encrypted filesystem/container or a comparably comprehensive virtual filesystem boundary, potentially combined with SQLCipher for database defence in depth. It does not prove which implementation is viable on Windows; that remains the prototype ticket's question.

## Required runtime tracing before architecture approval

Use a clean Windows VM and a uniquely identifiable canary corpus. Capture the Joplin process tree with Procmon (or equivalent ETW instrumentation), then diff files and scan allocated contents after each scenario:

1. cold start, profile creation, normal exit and forced termination during startup;
2. create/edit/delete a note, wait for history collection, use FTS and fuzzy search;
3. attach representative image/PDF/office files, OCR them, edit them externally and kill Joplin mid-edit;
4. enable semantic search/AI embeddings and inspect SQLite plus cache;
5. install a test plugin that writes settings, item user data, `dataDir`, installation dir, OS temp, home and arbitrary absolute paths;
6. sync through every retained target with E2EE forced, interrupt both upload and download, and inspect profile plus target temp paths;
7. invoke Backup, JEX/raw/HTML/PDF export, print, debug-report export and deletion-log export;
8. trigger renderer and main-process crashes with auto-upload both off and on; inspect `%LOCALAPPDATA%\CrashDumps`, Windows Error Reporting, minidumps and log attachments;
9. exercise multiple profiles, alternate instances, portable mode and `--profile`; verify the IPC secret and all early logs follow the intended boundary;
10. lock while plugins, sync, OCR, export and an external editor are active; confirm no canary survives outside approved plaintext egress locations.

Also inspect Windows pagefile/hibernation/crash-dump policy and filesystem-level remnants separately. Application encryption cannot guarantee absence from RAM snapshots, OS paging, antivirus quarantine, indexing, recent-file lists, thumbnail caches or third-party editor recovery files without platform controls. Those are threat-model decisions rather than Joplin profile files, but they must be named in the product's security limitations.

## Open blockers exposed by this inventory

- Does “encrypted at rest” mean only while Watchtower is closed/locked, or must files remain ciphertext while the app is open? A mounted whole-profile container satisfies the former, not the latter.
- Will v1 keep multi-profile support? If yes, is one unlock meant to expose all profiles or only one profile plus intentionally public launcher metadata?
- Are plaintext exports and external editing allowed as explicit user-authorised exceptions, or must they be removed from the Watchtower distribution?
- Will Watchtower retain crash uploading and the bundled Backup plugin? Stock behavior conflicts with the always-encrypted promise.
- What confinement, if any, is required beyond signature verification for curated plugins that can use Node/filesystem APIs?
- Which Windows storage primitive can meet crash consistency, update/recovery and cross-platform patchability without forcing a deep permanent divergence from Joplin upstream?

These questions should be answered by the encryption architecture, recovery and curated-plugin decision tickets; this report does not resolve them.
